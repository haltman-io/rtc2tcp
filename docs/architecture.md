# Architecture

## Binaries

- **`cmd/rtc2tcp-broker`** вҖ” runs the WebSocket broker and pairs peers by `rendezvous_token`.
- **`cmd/rtc2tcp-peer`** вҖ” runs either the `expose` or `connect` role.

## Internal packages

| Package                | Responsibility                                                                                                                     |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `internal/signaling`   | Broker wire messages and the WebSocket client used by peers.                                                                       |
| `internal/rendezvous`  | Broker implementation: pairing, session TTL janitor, per-IP rate limiter, trusted-proxy parsing.                                   |
| `internal/webrtc`      | Session state machine, WebRTC transport, control-channel wiring, transport-binding logic.                                          |
| `internal/auth`        | Interactive peer authentication (CPACE-Ristretto255 default, transitional ECDH retained for rollout compatibility only).           |
| `internal/tunnel`      | Bidirectional bridge between a TCP connection and a WebRTC DataChannel.                                                            |
| `internal/config`      | Flag parsing, connection-string parsing, build-time metadata (`Version`, `Commit`, `DefaultBrokerURL`).                            |
| `internal/logx`        | Structured, grep-friendly log formatter used by both binaries.                                                                     |
| `internal/banner`      | Version banner printed on startup.                                                                                                 |

The broker is only on the **signaling plane**. Once ICE completes and the DTLS-SRTP DataChannel is up, payload bytes flow peer-to-peer вҖ” typically direct, via a TURN relay when both peers are behind symmetric NATs.

## What the broker sees

- `rendezvous_token` (chosen by the operator or auto-generated by the expose side)
- Peer role (`expose` / `connect`)
- Session lifecycle events (`registered`, `paired`, `session_closed`, вҖҰ)
- SDP and ICE candidates during negotiation

## What the broker does **not** see

- TCP payload bytes
- DataChannel plaintext
- The `pairing_secret` вҖ” it stays on the two peers; the PAKE never sends it over the wire
- Anything after ICE completes вҖ” at that point the broker is just holding a WebSocket for late signalling
